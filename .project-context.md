# Контекст проекта VPS_VPN_PROXY

## Краткое описание
- Тип: Система развертывания прокси-сервисов на VPS
- Цель: Автоматическое развертывание HTTP и Telegram прокси
- Целевая платформа: Debian 11+ с Docker

## Архитектурные решения
1. **Контейнеризация**: Все сервисы в Docker
2. **Оркестрация**: Docker Compose для каждого сервиса отдельно
3. **Безопасность**: Fail2Ban + ограничение доступа
4. **SSL**: Автоматическое получение Let's Encrypt

## Ключевые компоненты
- `http-proxy/` - 3proxy с аутентификацией (порт 8080)
- `telegram-proxy/` - telemt с FakeTLS (порт 443), сайт-прикрытие на nginx (порт 80)
- `fail2ban/` - защита от атак
- `deploy.sh` - основной скрипт развертывания

## Критические файлы
- `deploy.sh` - главный скрипт (837 строк)
- `http-proxy/3proxy.cfg` - конфигурация HTTP прокси
- `telegram-proxy/telemt.toml` - конфигурация Telegram прокси
- `fail2ban/jail.local` - настройки Fail2Ban

## Известные проблемы, которые допускает автор
1. Безопасность: пароли в открытом виде в 3proxy.passwd - в виду того, что все операции выполняются от единственного пользователя root, прятать пароли не от кого
2. Конфигурация: отсутствие IP-фильтрации в 3proxy - не нужно, так как неизвестно, в каких приложениях будет подключен HTTP прокси
3. Версии: использование :latest тегов в Docker образах - автор надеется, что разработчики сервисов не изменят ключевых компонентов, но можно предложить зафиксировать мажорные версии
